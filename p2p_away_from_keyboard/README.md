## 开始扯淡

这是一条少有人走的路, 但既然实实在在的踩在了这条路上, 那么不仅要走的留下脚印, 还要走的更远. 我的这篇文章没有真正意义上的正文, 每一个小观点都可以彼此独立的阅读.

### 为什么叫 P2P 工作流

我想作为这本书的发起人, 我有必要在这个名称上先做一点解释, 以便我们对事物理解可以在这个点上保持尽可能的一致性.

当我们提到分布式办公, soho, remote 办公等名词时, 出发点往往是构成这样一个工作协作整体来看的, 说明的重点都不能很好的诠释构成这种办公体系中人与人的关系, 以及人该具有怎样的特质.

那么 P2P 网络具有怎样的特点呢?

- 每个节点地位平等
- 节点只有在网络中才能发挥其功效
- 网络一旦缺失全部节点就不复存在

那么这里对应到团队中, 便很容易的得到理想情况下人员结构及个人和团队的关系.

- 每个人在行政地位上平等
- 每个人在知识结构上要平等
- 个体需要团队才能更好的发挥作用
- 团队一旦丢失每个人便不复存在

上面的是理想情况下的特征. 其中更多表达着一种职能的扁平化和能力的同级化. 这里的能力不单单是个人的生产力, 还有个人的学习, 沟通, 表达, 理解的能力, 甚至更深入的是个人对自己生活规划和控制的能力.

在分布式办公这样简单粗暴的称呼之上, 通过思考后我认为叫做 P2P 办公更符合实际中这样团队的结构, 也更符合这样团队中每个人该有的特质的描述.

### P2P 办公不是不办公

为什么会说明这个点呢? 因为这里确实存在着一种奇怪的态度, 以至于不得不表达这一看起来理所应当的观点.

和朋友聊天时, 当听到我是在家办公便会不由的产生一生感慨: 好棒啊, 那你可以随便怎样都可以. 这样的表达我想是出于对这样办公方式了解不足产生的, 如果你自己亲身在经历 P2P 办公, 我想你一定不会这么感慨. 因为无论你的办公形式如何, 团队的结构如何, 最终每个人都是需要努力去作出自己该去做的那份工作, 而且正因为是在家里办公, 才更需要个人的自律性. 因为在这样的结构中, 每个人彼此物理上是隔离的, 但是却会因为工作流不同, 有时可能会更能明晰每个人在做的事情, 你的伙伴也更能看到你对团队的贡献是什么, 所以别说是不干活, 就算是延期这种事情都会更多的需要找寻之前评估时间时哪里出了问题. 也许在非远程办公中也会有同样的工作流, 但因为每天可以一起厮混, 有时闲聊的内容就从一个方面产生了在团队中的存在感, 而 P2P 办公中, 个人的存在感是需要工作量和工作成果来得以体现的.

当然我知道大部分朋友在说出这句话时都是由衷的在祝福我, 只是我比较矫情, 也自己想了这种微妙感觉背后的客观差别, 才特意想说明一下的.

### 办公从规划生活开始

生活这一词在很多 IT 从业者的脑海里都逐渐变的模糊了, 而 P2P 工作流中, 我认为这一概念是必须先被提出来并明确出来的. 常听到人们说要区分开工作和生活, 那么到底工作和生活的界限在哪里, 怎么区分, 是否真的需要区分, 如果区分了会带来怎样的结果呢.

我曾经在多个城市工作过, 北京, 珠海, 上海, 杭州. 每个城市有着各自的个性, 生活在其中会不由自主的被感染和影响, 但身处公司办公让我几乎错以为工作就是我的生活, 每天起来后想着就是去公司上班, 下班后回来累了就直接睡了, 即便是周末休息也并不懂得该如何度过, 沉浸在游戏, 美剧, coding 中. 生活仿佛除了工作一切都没有那么紧迫, 就如同面对乱七八糟的房间, 走出去上班反到让人感觉更充实, 却很少想着把一切规整的舒服些. 我想那时的自己一定是哪里病了, 不能真的看到自己的处境和工作外更为根基的困难.

我在工作后一直是个胖子, 不规律的饮食和睡眠把我带向更讨厌的自己, 一年体检时发现已经是轻度脂肪肝, 但我仍然没能意识到不单单是饮食的问题, 而是我的生活出了问题, 我把一切搞反了.

工作不是你的生活, 它本就该只是你生活中的一部分, 你用工作来换取一般购买力(钱), 同时为工作付出你相应的时间和自由. 这句话读起来不那么有撞击力, 但做起来却可能是阻碍重重的. 当我开始 P2P 办公的那天起, 我并没有意识到这样的工作方式会对自己的生活和自己对生活的理解产生如此大的影响. 即便有天我不再以 P2P 办公的方式参与工作, 我想这种理解也会帮我在自己的生活道路上走的更远.

所以我相信生活是需要规划的. 生活的规划最终就是对时间的划分和执行. 它不像切蛋糕, 有时你需要给自己更多的自由, 让自己去尝试, 宽容自己. 为什么这么说呢? 就减肥这件事来说, 我曾经尝试着以减掉体重为目的去锻炼, 去节食, 但压倒自己的稻草就是自己, 人们说坚持很难, 我觉得难得不是坚持, 而是如何选好一个决定的初衷. 有些初衷是具有爆发力的, 伴随着誓言和牺牲, 但如果更希望长远的养成一个习惯, 这个初衷就一定不能只有力量, 还要有更长远的预期. 所以这一次我便以增强心肺能力为初衷, 设定了一个期望的体重值, 开始了 Insanity 的锻炼. 每一次体能测试我都能看到自己心肺能力的提升, 这让我的初衷得到了积极的回应, 我信心满满的在继续坚持着, 体重随着我对饮食的控制也在降低, 最后减掉体重反成了一种初衷之外的事情, 是我这一决定的衍生品, 它会到来, 但我已不那么急迫的去追逐它. 我很难想象如果我最初就以减肥为目的, 今天是不是又在懊恼自己的身体状况了呢. 所以说作出选择的初衷是很重要的.

工作也是如此, 我们都需要更先明确这份需要你付出时间和自由的工作的初衷是什么. 如果是以更高工资为目的, 那么是必会带来不稳定不断的跳槽, 也许你喜欢也能从中得利, 但这种不稳定性外总需要有更多的追逐目标. 你可以看中一个公司的前途, 选择把自己捆绑在其中, 可以看中一个团队的 leader, 愿意没日没夜和他一起拼命. 无论你的选择是什么, 都不要忘了, 工作永远要带着你最初的预期, 一旦偏离了, 就需要及时的纠正.

P2P 工作的方式让我把自己从工作中剥离出来, 我看清了自己的血肉, 看清我所浸泡的杂乱的泥潭, 看清生活中更多的可能性.

### 前期过劳是必要的

说道过劳这一现象, 一般都会伴随着消极的声音和态度, 但在我看来, 在一个团队和项目的起步阶段, 必定是需要过劳的阶段, 它必不可少并且至关重要.

其实起初的 P2P 办公中, 我是伴随着过劳的. 这种过劳不是因为别人让我去干, 而是自己干着干着就已经深夜了, 这是对一种新的协作方式的适应过程, 身体和潜意识都条件反射性的对探究新鲜事物作出了极具爆发力的反馈. 这个阶段也是充满着愉快的, 并且基本可以看到自己在压榨自己的情况下可以到达怎样的极限. 我当然是极力的不希望你也经历这一切, 但我想这一切很可能是早期 P2P 团队中少不了的过程. 这个阶段中团队的每个成员都可能进入极限的过劳状态, 这就好像火箭升空前的加速推进, 没有这个阶段, 就很难在之后的轨迹上维持高速的前行.

### 争吵的意义

P2P 办公中, 因为物理距离的隔离和科技手段的局限, 常常不能像当面沟通的那么顺畅, 信息不对等的问题可能会时长出现. 对于这一问题, 单单依靠工具链是无法解决的, 至少现在是不行. 这里就需要更激烈的沟通方式. 我们的团队在沟通问题时, 很少有太过礼节的表述, 基本都是直截了当, 表达明确. 甚至有的时候会因为一些问题产生争吵, 我觉得这是好事, 首先说明了参与的个人是在乎这样的细节问题, 其次说明在争论的点上还没有达到执行层面的一致性. 此时争吵的意义就凸显出来.

我是一个不赞成凡事都和为贵的人, 我不认为违背个人意愿和观点的事情应该以 "和" 的形式掩盖过去. 而此处就是 P2P 工作流, 有别于传统转达 + 执行工作流的微妙之处, 人与人级别的因为更为扁平, 个人在团队关系网络中需要尽可能多接收其它人的知识还要尽可能传达自己的知识. 人不是机器, 接受过程就会产生冲突, 表现的形式就是争论甚至是争吵. 自己忽视的问题, 其他人很看重, 各自表达着观点的同时, 会带出观点背后的整套认知体系, 这会带动团队中不同角色的人有更多的可能性完成跨界的工作. 比如我们团队的前端工程师出了写 web 网页外, 还会自己用 go 写后端, 并且具有着产品思路, 在前端工作外还是可以独当一面的产品经理. 设计师不仅仅是手绘达人, 同时还为了更好的理解网页设计的理论体系, 自己写起代码. 我个人则由最初单纯的 iOS 开发开始熟悉更广泛的领域, Sketch 原型的绘制, koajs 撰写前后端, docker 部署代码, 嵌入式领域的开发等等, 这些都在因为物理上的隔绝而变的更为重要, 每个人都更需要知道自己伙伴所做东西背后的理论体系, 以便更好的配合完成工作.

这也正是叫做 P2P 办公，而不是 remote 办公的最重要的原因, 团队中的每个人都需要对和自己配合的伙伴所在的领域有所了解, 甚至可以一起完成同一性质的工作. 当然, 我并不是就此鼓励无意义的争吵, 抠字眼的争论反到会是不好的现象, 应该尽可能避免.

### 一次 P2P 办公的具体工作流

早上起来, 我会先准备早饭, 这是不能缺少的.

10 点钟团队会进行一次远程语音会议, 这里主要的工具有不少可选, 你可以选择 hangout, skype, qq. 我们需要明确每个人在昨天完成的事情以及今天要做的内容, 并不阔展开细节去讲, 而应该能够更概括性的说明白, 时间基本在每人五分钟的样子.

会后各自开始办公, 我特意将自己的生活和办公区域做了分离, 二楼办公, 一楼生活. 为了提升工作效率, 我不会在设备上做太多吝啬, 足够大的外接显示器可以带来更快的任务切换, 它是必不可少的. 我偏好 GitHub 的 Fork 模型来完成协同工作, 因为每一次修改都需要发起 Pull request 由其他一同工作的伙伴做 review, 这样不仅能知晓彼此工作的进度, 还可以知道具体的修改点. 有人会认为这样很低效, 但我认为在不知晓对方修改情况下的任何修改都可能带来重复的工作, 并需要额外的沟通来同步彼此的进度, 出了问题还是要去看代码, 这个过程反到更为低效.

午饭我基本都是自己和老婆来做, 很少出去吃.

一般下午会是工作比较集中的时间段, 再者就是晚上, 在不可避免的过劳阶段, 基本上凌晨两三点团队的大家都还在提代码.

这里需要再次强调的就是, 一切最终落实到设计和代码层面的东西, 都要尽一切可能和相关人一起讨论确认. 这里的工具没有那么高大上的东西, 我喜欢纸笔, 一方面是它让我用着踏实并且可以让我专注于内容本身, 另一方面它确实是我在表达图形化的东西时最快出活的东西. 确认需求, 并尽可能做力度的细分是很有必要的.

剩下的都是愉快的生活, 比如看个电影, 遛遛狗, 看看书, 跳跳 Insanity, 叫个朋友来家里彰显下厨艺.

但从工作流上来看, 这里没什么神秘的东西, 也没有什么特别不一样的, 但这里每个人都极其重要.

### 工具链

我想一定很多人关心工具链, 我知道这个很重要, 但是更重要的是团队每个人都要用起来才真的有意义. 之前在一个团队中, 有遇到 svn 管理设计稿的. 我倒不是说这样不可以, 而是违背了自己的一个基本理念: 用好的产品才能做出好的产品. 你让一个设计师用 svn 管理版本? 本来有更好的替代品为何不用呢? 还是意识上对新东西的接收速度太慢, 太过守旧. 直到离开那个团队时, 其他成员还有在用 baidu 作为默认搜索的, 我也是醉了.

言归正传说回工具链:

- 多人语音沟通
  - QQ: 你可以说这个太 low 了, 但是在实际的多人语音中, 它的表现确实不错
  - Skype: 语音表现很不错
  - Hangout: 不仅仅可以做语音沟通, 还可以共享屏幕来讨论需求
- 核心工具
  - Slack: 用来做工作沟通, 清晰的 channel 划分可以带来打扰性的最小化, 又不会遗失有效信息
  - Trello: 任务管理的细分都可以通过它来做, 虽然从功能上来说和 GitHub issues 有时有些重合, 但是却可以从更集中和宏观的层面将整个项目的情况反映出来
  - GitHub: 你可以选择自己搭建 Git 服务器, 但如果没有足够的理由我更偏好直接使用 GitHub, 可以直接通过 issues 标记里程碑并将细化的任务纪录其上, 提交即完成的方式快速明了, 无需再做事后的操作来标记完成. 同时在 P2P 工作流中 Fork 协作模型比分支协作模型更适合来做协作. 此外 GitHub 的 Application 也很丰富, 方便来完成代码部署和持续集成
- 文件共享
  - Dropbox: 说什么用百度网盘的都不要来沟通了
  - Copy: 这也是个不错的多人文件共享的服务, 但是总体来说还是没有 Dropbox 好用

建立在工具链之上还有很多习惯, 我个人还在不断学习, 深知还有太多的不足之处. 欢迎拍砖指出, 共同前行.
